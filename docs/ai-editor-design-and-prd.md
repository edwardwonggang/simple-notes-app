# AI 编辑器设计方案与需求文档

## 1. 先说清楚当前问题

你现在这个客户端，已经从“能跑”走到了“有产品雏形”的阶段，但还没有形成稳定架构。

目前最明显的问题不是某一个 bug，而是整体缺少约束：

- 编辑区的数据模型不稳定
- AI 问答和正文内容混在一起
- 文档的保存、删除、恢复规则不完整
- 设置项越来越多，但没有明确分类
- 样式和布局能显示，但没有统一设计规则
- 一个地方修好了，另一个地方又容易受影响

这很正常。因为你现在的实现更像“边写边长出来”，而不是“先定结构再实现”。

这类项目想继续往成熟方向走，下一步不能只靠修 bug，必须先定产品骨架。

---

## 2. 产品定位

建议你把这个产品定义为：

**本地优先的 AI 笔记编辑器**

不是纯聊天软件，也不是普通 Markdown 查看器，而是一个以“文档”为中心、AI 作为辅助能力的编辑工具。

这句话很重要，因为它决定了后面所有设计。

### 如果按聊天软件设计

- 中心对象是对话
- 文档只是附属结果

### 如果按 AI 编辑器设计

- 中心对象是文档
- AI 是文档编辑过程中的一个能力

建议你坚定选第二种。

## 4. 整体产品骨架

建议主界面固定为 4 个区域：

1. 左侧栏：文档与导航
2. 主编辑区：文档正文
3. 底部 AI 操作栏：提问、改写、生成
4. 设置页：应用配置、模型配置、外观配置、文件配置

不要再把 AI 输入做成“任意漂浮、任意插入位置”的主要交互模式。

原因很简单：

- 看起来高级，但实现难度很高
- 对初学者来说非常难稳定
- 很容易打乱编辑器结构

更稳妥的方案是：

- AI 主入口固定在底部
- 选区操作作为辅助入口
- AI 结果以块的形式插入文档

- 保存状态

### 5.3 左侧栏建议

从上到下排列：

- 新建文档按钮
- 搜索框
- 导航分组
- 文档列表
- 本地目录信息

- 正文最大宽度：780 到 900px

## 6. 编辑区应该怎样设计

编辑区是整个产品最重要的部分。

### 6.1 现在最大的问题

你目前的编辑区更像“直接改 DOM”，而不是“编辑结构化文档”。

这会导致：

- 回车行为不稳定
- AI 插入位置容易错
- 样式渲染容易混乱
- 保存的数据难以维护

### 6.2 正确方向：块结构

建议文档使用“块模型”来表示，而不是直接保存整页 HTML。

例如：

```ts
type DocumentBlock =
  | { id: string; type: 'paragraph'; text: string }
  | { id: string; type: 'heading'; level: 1 | 2 | 3; text: string }
  | { id: string; type: 'list'; style: 'bullet' | 'ordered'; items: string[] }
  | { id: string; type: 'quote'; text: string }
  | { id: string; type: 'code'; language: string; code: string }
  | { id: string; type: 'ai-question'; text: string }
  | { id: string; type: 'ai-answer'; format: 'markdown'; text: string }
  | { id: string; type: 'divider' }
  | { id: string; type: 'image'; src: string; alt?: string };
```

通俗解释：

- 保存时存的是块数据
- 展示时再把块数据渲染成 HTML
- 用户编辑时，本质上是在改块数据

这样后面会非常清晰：

- 删除某个 AI 回答只删一个块
- 重试某个回答只重跑一个块
- 排序只改块顺序
- 导出也容易做

### 6.3 编辑器交互规则

建议固定规则，不要模糊：

- `Enter`：新起一段
- `Shift + Enter`：当前段内换行
- `/`：打开插入命令菜单
- 选中文本后：可触发 AI 操作
- `Ctrl/Cmd + S`：立即保存

### 6.4 块级操作建议

每个块支持：

- 删除
- 复制
- 上移
- 下移
- 转换类型

例如：

- 段落 -> 标题
- 段落 -> 列表
- 段落 -> 引用

### 6.5 编辑区两种模式

建议后面支持：

- 编辑模式
- 专注模式

编辑模式：

- 工具更多
- 块操作按钮可见

专注模式：

- 侧栏弱化
- 顶栏弱化
- 当前块高亮
- AI 栏简化

这个思路可参考 Typora 的 Focus/Typewriter 模式。

## 7. AI 内容块怎样设计

AI 结果不要伪装成普通正文。

要区分，但不能太抢眼。

### 7.1 AI 问答块结构

建议成对出现：

- `ai-question`
- `ai-answer`

### 7.2 AI 问题块视觉

- 颜色轻微突出
- 不要太大
- 更像“指令”或“提示”

### 7.3 AI 回答块视觉

- 浅背景
- 与正文有一定区分
- 顶部有操作工具栏

### 7.4 AI 回答块工具栏

建议至少有：

- 复制
- 重试
- 插入正文
- 替换选区
- 折叠
- 删除

### 7.5 AI 的 3 个入口

建议提供：

1. 底部 AI 栏
2. 选中文本后的上下文操作
3. 块右侧的 `Ask AI`

### 7.6 AI 动作类型

建议分两类：

#### 直接问答

- 解释
- 总结
- 翻译
- 改写
- 续写

#### 文档操作

- 插入到当前位置
- 替换选中内容
- 在下方生成
- 生成为新文档

## 10. 背景与主题怎样设计

背景不要只做成“换颜色”。

建议分三层：

1. 应用背景
2. 编辑区背景
3. 内容卡片背景

### 10.1 纯净主题

- 背景轻微渐变
- 编辑区接近纯白
- 卡片白色

### 10.2 护眼主题

- 低饱和绿灰
- 编辑区浅暖绿
- 卡片更亮一点

### 10.3 纸张主题

- 暖灰外层
- 米白编辑区
- 轻纸张纹理

### 10.4 深色主题

- 应用背景深石墨
- 编辑区深灰蓝
- 卡片稍亮一层

### 10.5 背景设计原则

- 背景不能抢正文
- 不要每个块都加重阴影
- 不要用过多颜色
- 背景变化不能影响正文排版

## 11. 字体系统怎样设计

你一定要定一套字体和字号规则。

### 11.1 字体建议

正文：

- `Microsoft YaHei UI`
- `PingFang SC`
- `Segoe UI`
- `sans-serif`

代码：

- `Cascadia Code`
- `JetBrains Mono`
- `Consolas`
- `monospace`

### 11.2 字号令牌建议

- 文档标题：28
- 一级标题：24
- 二级标题：20
- 三级标题：18
- 正文：16
- 辅助文字：13
- 注释：12
- 代码：14

### 11.3 行高建议

- 正文：1.75 到 1.9
- 标题：1.3
- 列表：1.6 到 1.75
- 代码：1.6

### 11.4 粗细建议

只用 3 档：

- 400
- 500 / 550
- 700

不要随便让很多文字变黑体。

## 12. 文件怎样新建、删除、保存

这个部分必须做成规则，不要靠感觉。

### 12.1 文档生命周期

建议文档状态为：

- 新建未保存
- 已保存
- 已修改未保存
- 已删除进入回收站
- 已恢复
- 已彻底删除

### 12.2 新建文档

建议流程：

1. 点击新建
2. 立即生成文档对象
3. 默认标题为“未命名文稿”
4. 文档立即出现在左侧栏顶部
5. 焦点进入标题或正文

不要等用户输入后再创建文档对象。

### 12.3 保存文档

建议：

- 自动保存：500ms 到 1000ms 防抖
- 手动保存：`Ctrl + S`
- 保存状态明确显示

状态建议：

- 已保存
- 保存中
- 保存失败
- 未保存

### 12.4 删除文档

不要直接彻底删除。

建议：

- 删除 = 进回收站
- 回收站里再彻底删除

### 12.5 文件名与显示名分离

建议：

- 磁盘文件名使用 UUID
- 列表显示标题
- 标题可改
- 没有标题时取正文第一行

不要直接把标题当真实文件名。

## 13. 架构设计建议

### 13.1 当前问题

你现在更像是：

- 页面直接操作 DOM
- 然后顺便保存 HTML

这样做前期快，但后面会越来越难维护。

### 13.2 建议分 4 层

#### 领域层

定义文档、块、设置这些核心概念

#### 应用层

定义用户动作

例如：

- 创建文档
- 删除文档
- 插入块
- 更新块
- 发送 AI 请求

#### 基础设施层

定义如何存文件、调模型、导出、索引

#### UI 层

只负责显示和交互

### 13.3 推荐目录结构

```txt
src/
  core/
    document/
      document-types.ts
      document-service.ts
      document-commands.ts
    ai/
      ai-types.ts
      ai-service.ts
    settings/
      settings-types.ts
      settings-service.ts
    theme/
      theme-types.ts
  infrastructure/
    storage/
      documents-repo.ts
      settings-repo.ts
    ai/
      provider-client.ts
    export/
      export-html.ts
      export-pdf.ts
  ui/
    shell/
    sidebar/
    editor/
    ai-composer/
    settings/
  shared/
    utils/
    constants/
```

### 13.4 核心原则

以后一定尽量坚持：

- 数据是 JSON
- HTML 只是渲染结果
- CSS 只控制展示
- 所有用户动作都通过应用层命令修改数据

不要直接依赖“顺手改 DOM”。

## 15. 需求文档 PRD v1

## 15.1 产品名称

AI 本地笔记编辑器 Desktop

## 15.2 产品目标

帮助用户在本地文档中完成写作、整理、AI 生成、保存与管理，形成一个“文档优先”的 AI 编辑体验。

## 15.3 目标用户

- 主界面

### 文档

- 新建后 1 秒内出现在侧栏

### 编辑器

- `Enter` 稳定新起段落
- `Shift+Enter` 稳定换行
- 块间距一致
- AI 块不破坏正文布局

### AI

- 发送问题后固定追加到文档底部
- 问题块与回答块成对出现
- 失败时可重试
- 删除回答块不影响其它块

### 设置

- 所有修改有明确反馈
- 模型测试连接有结果提示

## 16. 开发顺序建议

你是开发小白，所以不要一口气做完。

### Phase 1：先稳底层

只做：

- 文档 JSON 模型
- 文档增删改查
- 自动保存
- 左侧栏文档列表
- 主编辑区仅支持基础块

目标：

先把产品骨架稳定下来。

### Phase 2：再做 AI 块

- AI 输入栏
- AI 回答块工具栏
- 重试
- 插入正文
- 删除 AI 回答

目标：

让 AI 变成稳定能力，而不是破坏编辑器的因素。

### Phase 3：再做设置和外观

- 设置页重构
- 主题与背景
- 字体与宽度控制
- 专注模式

### Phase 4：再做高级功能

- 搜索
- 收藏
- 回收站
- 模板
- 导出
- 快捷键

- 容易和正式内容混在一起
